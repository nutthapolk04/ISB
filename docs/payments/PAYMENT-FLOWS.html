A. วิธีจ่าย / เติมเงิน / ย้อนรายการ ทั้งหมดในระบบ

กลุ่มFlowสรุปสั้น
จ่ายที่ POS
(5 แบบ)		 1. Wallet / RFIDแตะบัตรสมาชิก ตัดเงินจากกระเป๋าภายในระบบ
2. Cashเงินสดหน้าร้าน คิดเงินทอนใน UI กระทบยอดตอน close-day

5. Departmentตัดกระเป๋าแผนก ติดลบได้ สถาบันจ่ายทีหลัง
เติมเงิน
(3 ช่องทาง)		 6a. Parent QR (bay_qr)ผู้ปกครองสแกน QR PromptPay จริงเติมกระเป๋าลูก
6b. Parent บัตร (bay_easypay)บัตรเครดิต/เดบิตผ่านหน้า EasyPay ของ BAY (+3% fee)

B. "เงินจริง" วิ่งที่ไหน

ที่อยู่ของเงินFlow ที่เกี่ยวลักษณะ
Internal ledger
(ตาราง wallet ในระบบ)		 Wallet/RFID, Department, Cashier top-up เป็นตัวเลขในฐานข้อมูลล้วน — เงินจริงเข้าระบบมาก่อนแล้ว (จากการเติมเงิน) หรือค้างเป็นหนี้แผนก
ธนาคาร BAY
(ผ่าน PYMT gateway)		 QR PromptPay ที่ POS, Parent top-up (bay_qr + bay_easypay) เงินจริงวิ่งเข้าบัญชีร้าน/สถาบันผ่านธนาคาร ระบบรู้ผลจาก webhook
เครื่อง EDC / acquirer
(Paywire)		 EDC ที่ POS เงินตัดที่ terminal และไป settle กับ acquirer — backend เก็บแค่หลักฐานอ้างอิง (approval code ฯลฯ)
เงินสดหน้าร้าน Cash, Cashier top-up อยู่ในลิ้นชัก กระทบยอดตอน close-day
จุดสำคัญที่สุดของภาพรวม
Gateway ภายนอกของจริงมีแค่ 2 ราย: (1) ธนาคาร BAY ผ่าน PYMT gateway (QR PromptPay + EasyPay) และ (2) Paywire / EDC acquirer (เครื่องรูดบัตร) — ที่เหลือทั้งหมดเป็น ledger ภายในหรือเงินสดหน้าร้าน

  1. แคชเชียร์เปิด RfidPaymentModal — hook useRfidListener รับ UID บัตรได้ 2 ทาง: WebSocket ws://localhost:9001 (RFID bridge) หรือ keyboard wedge (เครื่องอ่านพิมพ์ UID เหมือนคีย์บอร์ด)
  2. ระบบตามหาเจ้าของบัตรแบบ cascade: /customers/by-card → /users/by-card → by-code → by-username → departments (บัตรแผนก)
  3. ยิง POST /pos/checkout ด้วย payment_method: "wallet"
  4. pos_checkout_service.checkout() ตรวจ guard ตามลำดับ:
    • บัตรถูก freeze หรือไม่ (card-frozen guard)
    • daily limit ต่อ module — canteen default ฿500/วัน, store ฿25,000/วัน
    • เพดานติดลบ (negative credit limit) — เกินแล้วตอบ EXCEEDS_NEGATIVE_CREDIT_LIMIT
    • กระเป๋าของ user ห้ามติดลบ — เงินไม่พอตอบ INSUFFICIENT_USER_WALLET
  5. ผ่านทุก guard แล้วเขียนครบชุด: receipts + receipt_items + shop_movements (ตัด stock) + wallet_transactions (ชนิด DEDUCTION) + audit_logs + ส่ง low-balance alert ถ้ายอดเหลือน้อย
เงินใน flow นี้เป็น ledger ภายในล้วน — ไม่มีการเรียก gateway ภายนอกเลย

3QR PromptPay (BAY) — flow พิเศษสุด

หัวใจของ flow นี้
ใบเสร็จไม่ได้ถูกสร้างโดยแคชเชียร์ — ถูกสร้างโดย webhook จากธนาคาร เมื่อเงินเข้าจริงเท่านั้น ฝั่งหน้าจอทำได้แค่รอ (poll) จนกว่า intent จะเปลี่ยนสถานะ

→
POST /pos/qr-intent
payment_intents (pending)
→
PYMT gateway → BAY
ได้ qr_payload จริง
→
ลูกค้าสแกน
เงินเข้าธนาคาร
→
webhook /bay/callback
→ สร้างใบเสร็จ
  1. QrPaymentModal ยิง POST /pos/qr-intent พร้อม {amount, cart} → สร้างแถว payment_intents (intent_type: pos_sale, เก็บ cart_snapshot ทั้งตะกร้า, status pending)
  2. backend เรียกธนาคาร BAY ผ่าน PYMT gateway: 
    POST {PYMT_BASE_URL}/api/v1/bay/qr
x-config-app: bay.qrPayment
Authorization: Basic {PYMT_MERCHANT_TOKEN}
    ได้ qr_payload ของจริงกลับมาแสดงบนจอ
  3. ลูกค้าสแกนจ่ายผ่านแอปธนาคารของตัวเอง
  4. BAY ยิง webhook มาที่ POST /api/v1/bay/callback — backend ตรวจลายเซ็น HMAC-SHA256 จาก header x-pymt-signature ด้วย PYMT_WEBHOOK_SECRET
    ถ้าไม่ตั้ง secret ระบบจะรับ webhook แบบไม่ตรวจลายเซ็น — อนุญาตเฉพาะ dev เท่านั้น ห้ามปล่อยแบบนี้บน production
  5. handleBayCallback → confirmPosQrSale ทำงานแบบ idempotent 3 เฟส:
    • เฟส A: SELECT ... FOR UPDATE ล็อกแถว intent + claim ด้วย sentinel gateway_webhook_claimed + เงื่อนไข WHERE receipt_id IS NULL กันชนกันเอง
    • เฟส B: รัน checkout() ด้วย method "qr_promptpay" สร้างใบเสร็จจริงจาก cart_snapshot
    • เฟส C: ผูก receipt_id กลับเข้า intent + mark confirmed
  6. ระหว่างรอ frontend poll สถานะ intent ทุก 2 วินาที — ทุกรอบที่ 3 จะยิง /inquiry force-sync กับธนาคารด้วย, เพดานรอ 5 นาที, มีปุ่ม Check Now ให้เช็คเองได้ และกด cancel ได้เฉพาะตอน intent ยัง pending
Edge cases ที่ระบบกันไว้แล้ว
webhook ของ BAY เป็นแบบ at-least-once — มาซ้ำได้ — การ claim แบบ FOR UPDATE + sentinel + receipt_id IS NULL การันตีว่าใบเสร็จถูกสร้างครั้งเดียวไม่ว่า webhook จะยิงมากี่รอบ

Safety nets ของการเติมเงิน

  • /inquiry — เช็คสถานะกับธนาคารแบบ manual ได้ทุกเมื่อ
  • Reconcile sweep อัตโนมัติทุก 10 นาที — กวาด intent ที่ค้าง pending นานเกิน 15 นาที ไปถามธนาคาร; ถ้า BAY ตอบ 404 (ไม่รู้จักรายการ) = mark cancelled
อย่าสับสน mock กับของจริง
MockBayGateway / KrungsriGatewayDialog เป็น mock สำหรับ dev เท่านั้น — และ method credit_card / qr_promptpay (แบบเติมเงิน) เป็น legacy mock; ของจริงคือ bay_qr / bay_easypay ซึ่งจะตอบ 501/503 ถ้าไม่ได้ตั้ง PYMT_BASE_URL / PYMT_MERCHANT_TOKEN

  2. คืน stock กลับเข้าคลังแบบ FIFO
  3. คืนเงินเข้ากระเป๋าอัตโนมัติเฉพาะใบเสร็จ method WALLET / CARD_TAP — เขียนทั้งแถว REFUND และ VOID ใน wallet_transactions เพื่อคืนโควต้า daily limit ให้ด้วย
Method อื่นไม่คืนเงินอัตโนมัติ
cash / EDC / QR / department — void แล้วระบบคืนแค่ stock ส่วนเงินต้องคืนนอกระบบ (เงินสดคืนมือ, EDC ทำที่เครื่อง/acquirer)

ส่วน returns (คืนสินค้า) และ graduation refunds (คืนเงินคงเหลือตอนจบการศึกษา) เป็นอีกชุดหนึ่งแยกจาก void — อยู่ใน ReturnController / RefundController ไม่ลงรายละเอียดในเอกสารนี้

9หมายเหตุ Kiosk

Kiosk ไม่มี payment flow แยกของตัวเอง — เป็นเพียง device profile ที่เรียกใช้ endpoint ชุดเดียวกับ POS ทุกประการ ทุกอย่างในเอกสารนี้จึงครอบคลุม kiosk ด้วยโดยอัตโนมัติ
