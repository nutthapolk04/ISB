1ภาพรวมสถาปัตยกรรม

ระบบชำระเงินผ่าน EDC (card terminal) ของ ISB POS เชื่อมต่อผ่านห่วงโซ่ 4 ชั้น เริ่มจากหน้าเว็บ POS ไปจนถึงตัวเครื่อง EDC จริงที่หน้าร้าน:

หน้าเว็บ POS
React / Vite (frontend)
→
Vendored SDK client
frontend/src/lib/paywire/
→
Paywire bridge
HTTP + WebSocket
127.0.0.1:7331 หรือ
pos.local.bridge.schooney.tech:7331
→
เครื่อง EDC จริง
Verifone VTI / Newland LinkPOS

Bridge ที่อยู่ตรงกลางห่วงโซ่นี้มีอยู่ 2 แบบ ใช้แทนกันได้ตามสถานการณ์:

Bridge ตัวจริง — paywire.exe

แอป .NET 8 รันเฉพาะบน Windows x64 เท่านั้น ทำงานเป็น tray app บนเครื่อง POS พูดคุยกับเครื่อง EDC ด้วย protocol linkpos-bay-v1.05 และ vti-bay-aycap-v10.4.14 ขึ้นอยู่กับยี่ห้อเครื่อง

Bridge ตัวจำลอง — Mock bridge

Node server อยู่ที่ docs/edc/sdk-js/mock-bridge/server.js จำลอง protocol ของ bridge จริงครบทุกจุด ใช้พัฒนาบน Mac ได้โดยไม่ต้องมีเครื่อง EDC จริง โดยจำลองเป็นเครื่อง Newland N910 (LinkPOS) พร้อมจำลอง timing ของเครื่องจริง และอนุมัติรายการด้วย responseCode: "00" (ยกเว้นยอดลงท้าย 99 สตางค์ที่จำลองการปฏิเสธ — ดูหัวข้อ 4)

หมายเหตุ: ทั้งสอง bridge เปิด API รูปแบบเดียวกันทุกประการ (HTTP endpoints + WebSocket) ฝั่งหน้าเว็บ POS จึงไม่ต้องรู้เลยว่ากำลังคุยกับ bridge ตัวไหน — สลับได้แค่เปลี่ยนค่า env ตัวเดียว (ดูหัวข้อ 2 และ 4)

2การเชื่อมต่อทำงานยังไง

2.1 การเลือก scheme และ domain

EdcClient (ใน frontend/src/lib/paywire/client.ts) เลือก scheme ในการเชื่อมต่อโดยดูจาก protocol ของหน้าเว็บเองอัตโนมัติ:

  • หน้าเว็บเปิดด้วย http:// → ใช้ http:// + ws:// คุยกับ bridge
  • หน้าเว็บเปิดด้วย https:// → ใช้ https:// + wss:// คุยกับ bridge

ส่วน domain ของ bridge มาจาก environment variable VITE_EDC_BRIDGE_DOMAIN (ตั้งค่าใน frontend/.env) ถ้าไม่ได้ตั้งค่าไว้ ระบบจะใช้ค่า default เป็น pos.local.bridge.schooney.tech พอร์ต 7331 เสมอ

2.2 ขั้นตอนการเตรียมพร้อม (ready)

  1. เรียก GET /whoami ไปที่ bridge เพื่อขอข้อมูลตัวเอง จะได้ผลลัพธ์รูปแบบ:
{
  bridgeId: string,
  device: {
    brand: string,
    protocol: string,
    connected: boolean,
    capabilities: string[]
  },
  version: string
}
  1. จากนั้นเปิด WebSocket ที่ /status ค้างไว้ตลอดเพื่อฟังสถานะเครื่อง EDC แบบ real-time

2.3 สถานะการเชื่อมต่อ (WebSocket /status)

WS /status จะส่ง message รูปแบบ:

{ kind: "edc", edc: { state: "connected" | "disconnected", port, firmware, ... } }

modal EdcPaymentModal (frontend/src/pages/store/EdcPaymentModal.tsx) จะแปลงค่านี้เป็น pill สถานะบน UI:

● EDC connected   ● EDC not connected   ● กำลังเชื่อมต่อ (กะพริบ)

ถ้า WebSocket หลุดการเชื่อมต่อ ระบบจะพยายาม reconnect เองอัตโนมัติทุก 3 วินาที

2.4 การชำระเงิน (sale / qrSale)

เมื่อลูกค้าจะจ่ายเงิน modal จะเรียกเมธอดใดเมธอดหนึ่งของ SDK:

  • edc.sale() — สำหรับชำระด้วยบัตร (รูดบัตร/แตะบัตร)
  • edc.qrSale({ payment: "thaiqr" }) — สำหรับชำระด้วย Thai QR

ขั้นตอนภายในของ SDK เมื่อเรียกเมธอดใดเมธอดหนึ่ง:

  1. เปิด WebSocket ที่ /events?reqId=<idempotencyKey> เพื่อรับ mid-transaction events ระหว่างทาง (เช่น event qr-shown ตอนที่เครื่อง EDC แสดง QR code บนหน้าจอ)
  2. ยิง POST /txn/sale หรือ POST /txn/qrsale พร้อม header Idempotency-Key

SDK จะ stream mid-transaction events แบบ live ระหว่างที่ POST ยังค้างรอผลอยู่ — modal จึงขึ้นข้อความอย่าง "QR code is now shown on the terminal" ได้ทันทีที่ QR ขึ้นจอเครื่อง EDC จริง ๆ ไม่ใช่มาแสดงย้อนหลังหลังจบรายการเหมือนเวอร์ชันก่อน

Timeline ของรายการ QR โดยรวมจึงเป็นแบบนี้:

กดปุ่ม QR CODE
  → ยอดถูกส่งไปที่เครื่อง EDC ทันที
  → QR ขึ้นบนจอเครื่อง (event "qr-shown" → modal ขึ้นข้อความแจ้ง)
  → รอลูกค้าสแกน + ธนาคารตอบกลับ
  → อนุมัติ (responseCode "00" + approval code)
  → modal ขึ้น APPROVED แล้วบันทึกใบเสร็จอัตโนมัติ
ข้อควรระวัง: ผลสุดท้ายของรายการ (อนุมัติ/ปฏิเสธ) อยู่ใน HTTP response body ของ POST /txn/... เท่านั้น รูปแบบ { responseCode, approvalCode, fields } — ไม่ใช่ ข้อมูลที่มาจาก WebSocket โดย responseCode === "00" หมายถึงรายการได้รับการอนุมัติ

เมื่อเครื่อง EDC อนุมัติรายการ (responseCode === "00" และมี approval code ตอบกลับมา) modal จะ auto-confirm ทันที — เข้าสู่ step ใหม่ชื่อ "approved" แสดงกล่องสีเขียว APPROVED พร้อม spinner "Recording receipt…" แล้ว บันทึกใบเสร็จให้อัตโนมัติ โดยแคชเชียร์ไม่ต้องกรอกข้อมูลใด ๆ และไม่ต้องกดปุ่ม Confirm charge อีกต่อไป

ฟอร์มกรอกข้อมูลด้วยมือ (manual entry form) ยังคงมีอยู่ แต่เป็นเพียง fallback ใน 2 กรณีเท่านั้น:

  1. เชื่อมต่อ bridge ไม่ได้ (bridge unreachable)
  2. ได้ผลอนุมัติกลับมาแต่ไม่มี approval code แนบมาด้วย

2.5 หน่วยเงิน: บาท vs สตางค์

ระบบ ISB เก็บจำนวนเงินเป็น บาท (decimal) แต่ SDK ของ Paywire ต้องการหน่วยเป็น สตางค์ (integer) — modal จึงต้อง แปลงหน่วยก่อนส่งเสมอด้วยสูตร: 

const satang = Math.round(total * 100);

2.6 จุดที่เรียกใช้ modal

หน้า canteen (frontend/src/pages/Canteen.tsx) เปิด PaymentMethodPicker ให้ผู้ใช้เลือกวิธีชำระเงิน เมื่อเลือก EDC ระบบจะเปิด EdcPaymentModal — หน้า Store ก็ใช้ modal ตัวเดียวกันนี้ ร่วมกัน ไม่มีการแยกโค้ด modal คนละชุด

3สิ่งที่ทำไปเมื่อ 2026-07-16

  • Vendor mock bridge จาก ~/Downloads/sdk-js/mock-bridge เข้ามาไว้ใน repo ที่ docs/edc/sdk-js/mock-bridge/ (ประกอบด้วย server.js, package.json, package-lock.json โดยมี dependency เดียวคือ ws)
  • เพิ่ม VITE_EDC_BRIDGE_DOMAIN=127.0.0.1 ใน frontend/.env เพื่อให้ SDK ชี้ไปที่ mock bridge บนเครื่อง dev (Mac) แทน domain จริง
  • รีสตาร์ต Vite dev server (จำเป็น — Vite จะอ่านค่า env ตอน start เท่านั้น การแก้ .env ระหว่างรันอยู่จะไม่มีผล)
  • ตรวจสอบผลแล้ว: GET /whoami ตอบกลับ connected: true, WebSocket /status ส่ง state: "connected" ทันที → pill ใน modal ขึ้นเป็นสีเขียว ● EDC connected
  • ปรับ mock bridge ให้จำลอง timing จริง และปรับ SDK ให้ stream events แบบ live — ทดสอบ end-to-end แล้ว: qr-shown มาที่ +1.5s, อนุมัติที่ +10.1s, ยอด ฿10.99 ถูกปฏิเสธด้วย code 51 ตามที่ออกแบบ

4วิธีรัน/ทดสอบบนเครื่อง dev

ขั้นตอนแนะนำสำหรับพัฒนาและทดสอบบนเครื่อง dev (เช่น Mac) โดยไม่ต้องมีฮาร์ดแวร์ EDC จริง:

  1. รัน mock bridge

    ครั้งแรกให้ npm install ในโฟลเดอร์นั้นก่อน แล้วค่อยรันเซิร์ฟเวอร์:

    cd docs/edc/sdk-js/mock-bridge
npm install   # ครั้งแรกครั้งเดียว
node server.js

    เมื่อรันสำเร็จ mock bridge จะฟังอยู่ที่ http://127.0.0.1:7331

    Mock จำลอง timing ของเครื่องจริง (ปรับได้ผ่าน env): 

    # default: qrsale อนุมัติที่ ~10s, sale (บัตร) ที่ ~5s
MOCK_QR_DELAY_MS=10000 MOCK_SALE_DELAY_MS=5000 node server.js

# อยากให้เร็วขึ้นตอน dev เช่น
MOCK_QR_DELAY_MS=3000 node server.js

    ระหว่างทำรายการ terminal ที่รัน mock จะ log เหตุการณ์เป็นขั้น ๆ ให้ดูตาม (showing QR → waiting for scan + bank → approved/declined)

  2. ตรวจ env ของ frontend

    เช็คว่า frontend/.env มีบรรทัดนี้อยู่:

    VITE_EDC_BRIDGE_DOMAIN=127.0.0.1
  3. รัน dev server ของ frontend
    cd frontend
bun run dev

    ถ้าเพิ่งแก้ .env ต้องรีสตาร์ต dev server ใหม่เสมอ

  4. ทดสอบบน UI

    เปิดหน้า canteen → กดชำระเงิน → เลือกวิธีชำระเป็น EDC → เช็คว่า pill ขึ้นสีเขียว → กดปุ่ม QR CODE หรือ CREDIT CARD → ยอดถูกส่งไปที่ mock bridge ทันที

    Timing ที่จะเห็นจริง (ค่า default):

    • QR CODE: ~1.5 วินาที modal ขึ้นข้อความ "QR code is now shown on the terminal" (event qr-shown) → รอจำลองลูกค้าสแกน + ธนาคารตอบ → อนุมัติที่ ~10 วินาที
    • CREDIT CARD: อนุมัติที่ ~5 วินาที

    เมื่ออนุมัติ modal จะขึ้น APPROVED แล้วบันทึกใบเสร็จให้อัตโนมัติ (ไม่ต้องกรอกข้อมูล ไม่ต้องกด Confirm) → เห็น ReceiptSuccessModal เป็นอันจบรายการ

  5. ทดสอบกรณีปฏิเสธ (declined)

    ใช้ยอดที่ลงท้าย 99 สตางค์ เช่น ฿10.99 — mock จะจำลองธนาคารปฏิเสธด้วย code 51 INSUFFICIENT FUNDS ใช้ทดสอบหน้าจอ DECLINED และปุ่ม Try again ได้

  6. ทดสอบด้วย curl (เสริม)
    curl http://127.0.0.1:7331/whoami

5การใช้งานบน POS จริง (Windows) — Production notes

  • ลบหรือคอมเมนต์บรรทัด VITE_EDC_BRIDGE_DOMAIN ออกจาก .env เพื่อกลับไปใช้ค่า default pos.local.bridge.schooney.tech
  • ติดตั้ง paywire.exe (จากชุด Paywire_1.0.0) และ EDC USB driver (whql_Driver2020.zip) บนเครื่อง POS ที่รัน Windows
  • paywire.exe มี mock mode ในตัวเองด้วย สามารถรันแบบไม่มีฮาร์ดแวร์จริงได้: 
    paywire --mode=mock
  • หน้าเว็บที่เสิร์ฟผ่าน HTTPS จะคุยกับ bridge แบบ https/wss — bridge ต้องมี certificate (wildcard cert) ติดตั้งไว้ และถ้าเว็บมี CSP (Content-Security-Policy) ต้องเพิ่ม connect-src ให้ครอบคลุม domain:7331 ของ bridge ด้วย
  • สามารถทดสอบจากเครื่อง POS จริงได้ผ่าน hosted test Web POS ที่: https://isb-pos.schooney.tech

งานที่ยังเหลือ (roadmap)

ลำดับงานที่ยังต้องทำต่อ:

  1. เพิ่มคอลัมน์ในฐานข้อมูล: rrn, invoice_no, payer_id, idempotency_key, payment_selector, response_code
  2. ทำ void gating (รายการ Thai QR / QR-card ห้าม void)
  3. ทำ void/refund จริง (ของจริง ไม่ใช่ mock)
  4. ทำ settle สิ้นวัน

6อภิธานศัพท์ (Glossary)

คำศัพท์ที่ใช้บ่อยในระบบ EDC integration และความหมายโดยย่อ
ศัพท์ ความหมาย
EDCย่อจาก Electronic Data Capture คือเครื่องรูด/แตะบัตรชำระเงินที่หน้าร้าน
Bridge (paywire.exe)โปรแกรมตัวกลาง (.NET 8, tray app) ที่รันบนเครื่อง POS Windows เพื่อคุยกับเครื่อง EDC จริงผ่าน protocol เฉพาะยี่ห้อ
SDK (@paywire/sdk-js)ไลบรารี JavaScript ฝั่ง frontend ที่ห่อการเรียก HTTP/WebSocket ไปยัง bridge ให้ใช้งานง่าย
Mock bridgeเซิร์ฟเวอร์ Node.js จำลองพฤติกรรมของ bridge จริงทั้งหมด ใช้พัฒนา/ทดสอบโดยไม่ต้องมีเครื่อง EDC
whoamiHTTP endpoint GET /whoami ที่ bridge ใช้บอกข้อมูลตัวเองและสถานะเครื่อง EDC ที่ต่ออยู่
WebSocket /statusช่องทาง WS ที่ bridge ใช้แจ้งสถานะการเชื่อมต่อของเครื่อง EDC แบบ real-time
/eventsช่องทาง WS ที่ใช้รับ mid-transaction events ระหว่างทำรายการ เช่นตอนแสดง QR code บนจอเครื่อง EDC
Idempotency-KeyHTTP header ที่ใช้กันการทำรายการซ้ำซ้อนโดยไม่ตั้งใจ ค่าคือ reqId เดียวกับที่ใช้เปิด WS /events
responseCode ("00")รหัสผลลัพธ์รายการที่ได้จาก HTTP response ของ /txn/... ค่า "00" หมายถึงอนุมัติสำเร็จ
Approval codeรหัสอนุมัติรายการที่ธนาคาร/ผู้ให้บริการชำระเงินออกให้เมื่อรายการสำเร็จ — ระบบใช้บันทึกใบเสร็จอัตโนมัติ ถ้าไม่มีค่านี้จึงจะเปิดฟอร์มกรอกด้วยมือ
Auto-confirmการบันทึกใบเสร็จอัตโนมัติทันทีหลังเครื่อง EDC อนุมัติรายการ (responseCode "00" + approval code) โดยไม่ต้องกรอกหรือกดยืนยันด้วยมือ
RRNRetrieval Reference Number รหัสอ้างอิงรายการที่ใช้ตรวจสอบหรือกระทบยอดในภายหลัง
Masked PAN / payer_idเลขบัตรที่ถูกปิดบางส่วน (เช่น 4-4-4) เพื่อความปลอดภัย ใช้ระบุตัวผู้จ่ายโดยไม่เก็บเลขบัตรเต็ม
Invoice noเลขที่ใบกำกับ/รายการที่เครื่อง EDC หรือระบบ POS ออกให้ต่อธุรกรรม
สตางค์ (satang) vs บาทหน่วยเงินไทย — ระบบ ISB เก็บเป็นบาท (decimal) แต่ SDK ต้องการสตางค์ (integer) ต้องคูณ 100 ก่อนส่ง
LinkPOSProtocol ที่ใช้คุยกับเครื่อง EDC ยี่ห้อ Newland (เช่นรุ่น N910)
VTIProtocol ที่ใช้คุยกับเครื่อง EDC ยี่ห้อ Verifone
Capabilityความสามารถของเครื่อง EDC ที่ bridge ประกาศไว้ใน whoami เช่นรองรับ qrSale.thaiqr หรือไม่
qrSale / walletSaleเมธอดของ SDK สำหรับทำรายการชำระผ่าน QR code หรือ e-wallet ตามลำดับ
Voidการยกเลิกรายการที่เพิ่งทำสำเร็จ (ยังไม่ตัดยอดจริง) — บางประเภทการชำระ เช่น Thai QR ห้าม void
