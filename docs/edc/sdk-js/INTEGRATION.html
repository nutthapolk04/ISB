@paywire/sdk-js · Integration · VTI (Verifone) + LinkPOS (Newland) · v0.1

Connect your Web POS to the terminal

Paywire runs on each till PC and bridges your cloud Web POS to the local EDC. This SDK is the browser-side client. The same calls work whether the till has a Verifone VTI or a Newland LinkPOS terminal — the bridge maps each call to the device's wire protocol. You feature-detect; you don't branch on brand.

01Install & connect

import { EdcClient } from "@paywire/sdk-js";

const edc = new EdcClient({
  domain: "bridge.schooney.tech",   // production; or "127.0.0.1" for local mock testing
  acceptedDevices: [
    { brand: "newland",  protocol: "linkpos-bay-v1.05" },
    { brand: "verifone", protocol: "vti-bay-aycap-v10.4.14" },
  ],
});
await edc.ready();                  // throws UnsupportedDeviceError if the paired device isn't accepted

edc.bridgeId;          // "POS-N910"  (per-till id; never shown to the cashier)
edc.device;            // { brand, model, protocol, firmware, connected, capabilities }
edc.capabilities;      // ["sale","qrSale.thaiqr","walletSale.alipay","query","settle", …]
The golden ruleBuild your UI from edc.capabilities, not from the brand. A method whose capability is missing throws UnsupportedCapabilityError synchronously, before it ever reaches the terminal — so feature-detect and hide what isn't supported.

03Commands

Every command is an async generator yielding events; the terminal one is always kind:"result". Always pass a unique idempotencyKey.

// Card sale
for await (const ev of edc.sale({ amount: 10000, idempotencyKey: key, posRef })) {
  if (ev.kind === "result") { /* ev.responseCode, ev.approvalCode, ev.payerId, ev.responseMessage */ }
}

// QR / wallet — canonical selector; the bridge maps it to the device wire value
await edc.qrSale({ amount: 5000, idempotencyKey: key, payment: "thaiqr" });
await edc.walletSale({ amount: 5000, idempotencyKey: key, payment: "alipay" });

// Management
await edc.void({ posRef: lastPosRef, idempotencyKey: key });     // or { invoice: "000001" } on VTI
await edc.refund({ amount: 5000, posRef, idempotencyKey: key });
await edc.settle();
await edc.commsTest();

// LinkPOS-only (capability-gated) — see §05
await edc.query({ posRef: lastPosRef, idempotencyKey: key });    // timeout recovery
await edc.verify({ posRef: lastPosRef, payment: "thaiqr", idempotencyKey: key });
No events on LinkPOSNewland terminals don't stream chip-inserted/pin-required — the customer interacts with the terminal directly. Show a "Follow the prompts on the terminal…" status and wait for the final result. Code that only reads result works on both device families.
Void is cardholder-present AND payment-type-limited VOID/REFUND require a tap/confirm on the terminal — treat them like a sale (long timeout, "follow the prompts…"). A void needs the original sale's invoice + approvalCode (retain them from the SALE result); the terminal TID is not required. Crucially, only Card and local e-wallets (Alipay, WeChat, TrueMoney, LINE Pay, AirPay, Dolfin) can be voided by command. Thai QR, QR Visa and QR Mastercard CANNOT be voided via the API — they must be voided on the terminal itself. Gate your Void button by the last sale's payment type (the examples/web-pos disables Void after a Thai QR sale). Card REFUND may return NS "Transaction is Not Support" when refunds aren't enabled on the terminal/merchant profile.

05Idempotency & timeout recovery

  • Retry with the same key. On a network blip, replay the same idempotencyKey. The EDC returns DR (LinkPOS) or DI (VTI) and you surface the prior result as success — no double charge. New transaction → new key.
  • LinkPOS query. If a sale times out, don't re-charge blindly. Call query({ posRef }): 00 means it went through (show the receipt); NE means it never happened (safe to retry).
  • LinkPOS verify. For Alipay/WeChat/Thai-QR, if the customer insists they paid, verify({ posRef, payment }) asks the EDC to re-check with the acquirer host (it opens its camera to scan the QR/receipt).

06Local development — no certissuer, no TLS

✓ Certificates are a production-only concernFor local testing you need no certissuer and no TLS. The SDK uses http:///ws:// automatically when your page is served over plain HTTP, and the bridge runs plain HTTP on 127.0.0.1:7331 in mock mode.
# 1. Run the bridge against the in-process mock (no hardware, no certs):
cd paywire
dotnet run --project src/Paywire.Host -- --config paywire.linkpos.mock.yaml --mode mock --no-tray

# 2. Build the SDK and serve the example Web POS:
cd sdk-js && npm install && npm run build
cd examples/web-pos && npm install && npm run dev      # serves on http://127.0.0.1:5173

# 3. Open the page → EdcClient({ domain: "127.0.0.1" }) → ring up a sale.

The only non-cert prerequisite: the bridge's CORS allowlist (listen.cors_origins in the YAML) must include your dev origin — the sample config already lists http://localhost:5173 and http://127.0.0.1:5173.

07Hard rules (PCI & safety)

  • Never log or display full PAN, track2, CVV, PIN, or cardholder name. maskedPan / payerId are the only card-ish values, and both are already masked.
  • Never build the bridge URL by hand — use new EdcClient({ domain }). Hard-coding 127.0.0.1 breaks production (the real hostname resolves to loopback on each till via public DNS).
  • Never hard-code bridgeId — read it from /whoami.
  • Always pass idempotencyKey and retry with the same key.
  • Feature-detect with edc.capabilities before showing buttons.
