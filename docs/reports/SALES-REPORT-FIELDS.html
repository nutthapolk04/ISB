Daily Sales Report — ความหมายฟิลด์ amt_*

อ้างอิงความหมายของทุกฟิลด์เงินใน Sales Summary / Daily Sales Report เพื่อไม่ให้ตีความผิดซ้ำ (แก้บั๊กจริงเมื่อ 2026-07-21)

สารบัญ
  1. ภาพรวม
  2. ตารางฟิลด์ทั้งหมด
  3. Payment Method → Bucket ไหน
  4. ตัวอย่างจริง (เงินสด 138 / รับ 200 / ทอน 62)
  5. Grand Total คำนวณยังไง
  6. ประวัติการแก้ไข (2026-07-21)
  7. ที่ตั้งโค้ด
  8. Sales by Payment Report — Export Grouping

1ภาพรวม

รายงาน Daily Sales Report (หน้า /canteen/reports และ /store/reports, type salesSummaryReport) ดึงจาก endpoint GET /reports/sales-summary — คำนวณสด (ไม่ใช่ column ใน DB) จาก receipts.total และ receipts.cash_received ต่อใบเสร็จ 1 แถว

สำคัญ: ชื่อฟิลด์บางตัวอ่านผิวเผินแล้วเข้าใจผิดได้ง่าย โดยเฉพาะ amt_billing ที่ ไม่ใช่ "ยอดบิลของ DEPARTMENT" อีกต่อไป (เปลี่ยนความหมายไปแล้วเมื่อ 2026-07-21 — ดูหัวข้อ 6) อ่านตารางหัวข้อ 2 ให้ครบก่อนแก้โค้ดหรืออ่านตัวเลขจากรายงานนี้

2ตารางฟิลด์ทั้งหมด

Field ความหมาย สูตร / ที่มา
amt_receive เงินที่ลูกค้าจ่ายให้จริง (gross) — สำหรับเงินสดคือยอดที่แคชเชียร์พิมพ์ใน "เงินที่รับมา" อาจมากกว่ายอดบิลเพราะมีเงินทอน CASH: cash_received
วิธีอื่น: = amt_billing (จ่ายเป๊ะ ไม่มีทอน)
amt_change เงินทอนที่ให้ลูกค้ากลับไป CASH เท่านั้น: max(cash_received − amt_billing, 0)
วิธีอื่น: 0 เสมอ
amt_billing ยอดบิล/ยอดขายจริงของใบเสร็จนี้ (revenue) — เท่ากันทุกวิธีจ่ายเงิน ไม่ใช่ bucket เฉพาะวิธีใดวิธีหนึ่ง = amt_receive − amt_change
(= receipts.total เสมอ)
amt_cash ส่วนของ amt_billing ที่จ่ายด้วย เงินสด = amt_billing ถ้า payment_method = CASH, ไม่งั้น 0
amt_campus_card ส่วนที่จ่ายด้วย Campus Card (Wallet) = amt_billing ถ้า payment_method = WALLET / CARD_TAP, ไม่งั้น 0
amt_credit_card ส่วนที่จ่ายด้วย บัตรเครดิต/เดบิต/EDC = amt_billing ถ้า payment_method = CREDIT_CARD / DEBIT_CARD / EDC, ไม่งั้น 0
amt_qr_code ส่วนที่จ่ายด้วย QR Code / โอนเงิน = amt_billing ถ้า payment_method = BANK_TRANSFER, ไม่งั้น 0
amt_department ส่วนที่ เรียกเก็บหน่วยงาน แทนลูกค้าจ่ายเอง (Department Billing) = amt_billing ถ้า payment_method = DEPARTMENT, ไม่งั้น 0
amt_other ส่วนที่จ่ายด้วยวิธีอื่นนอกเหนือจากด้านบน (ไม่ค่อยเกิดขึ้นจริง) = amt_billing ถ้า payment_method = OTHER, ไม่งั้น 0

ผลรวมของ 6 field สุดท้าย (amt_cash + amt_campus_card + amt_credit_card + amt_qr_code + amt_department + amt_other) เท่ากับ amt_billing เสมอ — เพราะแต่ละใบเสร็จจ่ายด้วยวิธีเดียว มีแค่ 1 ใน 6 ช่องที่ไม่เป็นศูนย์

4ตัวอย่างจริง (เงินสด 138 / รับ 200 / ทอน 62)

สินค้าราคา ฿138 จ่ายเงินสด ลูกค้ายื่นแบงก์ ฿200 ทอน ฿62:

amt_receive200.00
amt_change62.00
amt_billing138.00 (= 200 − 62)
amt_cash138.00
amt_campus_card / amt_credit_card / amt_qr_code / amt_department / amt_other0.00
ก่อนแก้ (บั๊กเดิม): amt_receive ผูกกับ receipts.total เสมอ (ไม่สนใจ cash_received) ทำให้ amt_receive = 138 (ผิด ควรเป็น 200) และ amt_billing ตอนนั้นยังเป็น bucket เฉพาะ DEPARTMENT เท่านั้น จึงโชว์ 0 สำหรับการขายเงินสดทุกใบ (คนละความหมายกับตอนนี้)

5Grand Total คำนวณยังไง

"Grand total" บนหัวหน้า Daily Sales Report = sum ของ amt_billing ทุกแถว (totals.amt_billing) — คือยอดขายสุทธิจริงของวันนั้น ไม่ใช่ sum ของ amt_receive

ห้ามเอา totals.amt_receive มาโชว์เป็น Grand Total — เพราะสำหรับเงินสด amt_receive รวมเงินทอนเข้าไปด้วย ยอดรวมทั้งวันจะพองสูงกว่ายอดขายจริงเท่ากับเงินทอนสะสมทั้งวัน

แถวย่อยต่อร้าน (multi-vendor subtotal) และ export PDF/Excel ใช้หลักการ เดียวกัน — sum ทุก field แยกตามร้าน แล้วให้ผลรวม 6 bucket เท่ากับ amt_billing ของร้านนั้นเสมอ

6ประวัติการแก้ไข (2026-07-21)

  1. อาการที่พบ

    ขายเงินสด ฿138 รับเงิน ฿200 ทอน ฿62 — DB/API ตอบ amt_billing: 0, amt_receive: 138 ทั้งที่ควรเป็น amt_billing: 138, amt_receive: 200

  2. สาเหตุ

    amt_receive ผูกกับ receipts.total เสมอ ไม่เคยอ่าน receipts.cash_received เลย (ใช้แค่คำนวณ amt_change แล้วทิ้ง) และ amt_billing (ตอนนั้น) เป็น bucket เฉพาะ payment_method = DEPARTMENT เท่านั้น จึงเป็น 0 สำหรับการขายเงินสดโดยธรรมชาติ — ไม่ใช่บั๊กในความหมายเดิม แต่ชื่อฟิลด์ทำให้เข้าใจผิดว่าควรเป็นยอดบิลทั่วไป

  3. การแก้ไข
    • amt_receive เปลี่ยนเป็นเงินที่รับมาจริง (gross tendered) — อ่านจาก cash_received สำหรับ CASH
    • amt_billing เปลี่ยนความหมายเป็นยอดบิลทั่วไป = amt_receive − amt_change (เท่ากับ receipts.total) ทุกวิธีจ่ายเงิน
    • เพิ่ม field ใหม่ amt_department รับหน้าที่ bucket ของ DEPARTMENT แทน amt_billing เดิม
    • แก้ label หน้าเว็บ "Amt. Department" ที่เคยผูกผิดกับ amt_other ให้ผูกกับ amt_department
    • Grand Total หน้ารายงานเปลี่ยนไปอ้าง totals.amt_billing แทน totals.amt_receive

8Sales by Payment Report — Export Grouping (2026-07-21)

รายงานคนละตัวกับ Daily Sales Report ด้านบน — Sales by Payment Report (GET /reports/sales-by-payment) สรุปยอดขาย แยกตามวิธีจ่ายเงิน ไม่ใช่แยกตามใบเสร็จ export PDF/Excel ของรายงานนี้เจอบั๊ก + ปรับโครงสร้างเมื่อ 2026-07-21 เช่นกัน

อาการที่พบ

คอลัมน์ "Payment Method" ในไฟล์ export โชว์ raw key ดิบ เช่น payment.cash, payment.wallet, payment.edc แทนที่จะเป็นชื่อที่อ่านออก

สาเหตุ

โค้ดเดิมเรียก t(`payment.${method}`) โดยหวังว่ามี i18n key ชื่อ payment.cash ที่ root แต่จริงๆ key ชุดนี้ซ้อนอยู่ใต้ audit.payment.* เท่านั้น (ใช้สำหรับ audit log โดยเฉพาะ) — ไม่มี key ชื่อ payment ที่ root เลย เมื่อ i18next หา key ไม่เจอ จะคืน key string ดิบกลับมาแทน ("payment.cash") ซึ่งเป็นค่าที่ไม่ว่าง เลยไม่ไปเข้า fallback || r.payment_method ที่เขียนไว้

การแก้ไข

  • เลิกพึ่ง i18n สำหรับคอลัมน์นี้ ใช้ label map ตรงๆ ในหน้า Reports.tsx แทน (รายงานนี้เป็นภาษาอังกฤษล้วนอยู่แล้ว)
  • WALLET / CARD_TAP → "Campus Card" (เดิมโชว์ "Wallet")
  • CREDIT_CARD / DEBIT_CARD / EDC → "Credit Card"
  • BANK_TRANSFER → "QR Code", DEPARTMENT → "Department Use", อื่นๆ → "Other"
  • โครงสร้าง export เปลี่ยนจาก "retail block + department subsection" เป็น group ตามวิธีจ่ายเงิน — หัวข้อ section ต่อวิธี → แถว ACTIVE/VOIDED ต่อวิธี → แถว "Total <วิธี>" สรุปยอด (แสดงเฉพาะวิธีที่มีข้อมูลจริง)
  • raw payment_method หลายค่าที่ควรโชว์บัคเก็ตเดียวกัน (เช่น WALLET + CARD_TAP) จะรวมเป็นแถวเดียวต่อ status ไม่แยกซ้ำ
