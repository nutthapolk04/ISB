🔍 สถานะปัจจุบัน — Code Audit

✅ มีแล้ว / ทำถูกต้องแล้ว

  • Increment + row lock ครบทุกจุดที่แตะยอดเงิน — wallet_service.ts, topup_service.ts, pos_checkout_service.ts, pos_service.ts, refund_service.ts, returns_service.ts ทุกจุดใช้ pgClient.begin() + SELECT ... FOR UPDATE คำนวณ balanceAfter = balanceBefore ± amount เสมอ ไม่มีจุดไหน set ยอดทับตรงๆ
  • Timeout บน outbound request ที่มีอยู่ทั้งหมด — pymt_gateway.ts (BAY) 30s, auth_service.ts (Google OAuth) 10s — ไม่มีจุดไหนยิง fetch แบบไม่ตั้ง timeout
  • Rate limit ระดับ IP+path มีอยู่แล้ว — global 300 req/min, auth endpoint 30 req/min — ตรงกับเลข baseline ที่ระดมไอเดียไว้พอดี
  • Request timing + request-id logging — log START/END ทุก request, log แยกไฟล์รายวัน เก็บ 30 วัน
  • Webhook idempotency พื้นฐาน — เช็ค status === 'confirmed' ก่อนเสมอ ไม่ credit ซ้ำ

❌ ยังไม่มี / เป็นช่องโหว่จริง

  • ไม่มี process-level crash guard — ไม่มี uncaughtException/unhandledRejection handler เลย error จุดเดียวฆ่าทั้ง process ได้
  • ไม่มี CPU/Memory monitoring — ไม่มี metric endpoint ให้ดู resource จริง
  • ยังไม่มี deploy spec เป็นลายลักษณ์อักษร
  • ไม่มี VA scan / SQLi audit อย่างเป็นทางการ (โครงสร้างปลอดภัยระดับหนึ่งจาก ORM parameterized)
  • ไม่มี automated test สำหรับ flow การเงินเลย — จุดเสี่ยงสูงสุดถ้าจะรีแฟคเตอร์ต่อ
  • ไม่มี load test script ในโปรเจกต์
  • ไม่มี incident report template
  • Logging ยังเป็น file/console ล้วน — ไม่มี central dashboard/APM

🔐 1. Security

  • ยังไม่ทำ ทำ VA scan (Vulnerability Assessment) ทั้งระบบ (API, dependencies, container image)
  • ยังไม่ทำ ตรวจสอบ SQL injection อย่างเป็นทางการ (โครงสร้างปลอดภัยอยู่แล้ว — ORM parameterized ทั้งหมด — แต่ยังไม่เคย scan ยืนยัน)
  • ยังไม่ทำ กำหนด SLA: ทุก API ต้องตอบกลับภายใน 1 วินาที ภายใต้โหลดปกติ (มี timing log อยู่แล้ว รอแค่เกณฑ์วัดผลอัตโนมัติ)

💡 บทเรียนจากรอบนี้: การวางแผน "จะทดสอบยังไง" แล้วเขียน test จริงเจอบั๊กที่ใหญ่กว่าที่ประเมินจากการอ่านโค้ดอย่างเดียว (severity เปลี่ยนจาก Medium เป็น Critical) — ยืนยันว่าหมวด 6 (ทำ test case ให้ครบ) ควรทำเร็วๆ นี้ ไม่ใช่แค่หมวด security

💰 2. ความถูกต้องของธุรกรรมการเงิน (เติมเงิน/หักเงิน)

  • เสร็จแล้ว เติมเงิน/หักเงินต้องใช้ increment (บวก/ลบยอด) ห้าม set ยอดใหม่ทับยอดปัจจุบัน — ทำถูกต้องแล้วทุกจุด
  • เสร็จแล้ว Lock resource ระหว่างอัปเดตยอด — ใช้ SELECT ... FOR UPDATE ในทุก transaction แล้ว
  • ยังไม่ทำ ยังขาด automated test ที่ยิง concurrent request ซ้อนกันจริงเพื่อยืนยันว่า lock ทำงานถูกต้อง (ตอนนี้ verify ได้แค่จากอ่านโค้ด)

🛡️ 3. ความทนทานของระบบ (Resilience)

  • เสร็จแล้ว ทุก outbound request ต้องมี timeout — มีครบทุกจุด (BAY gateway 30s, Google OAuth 10s)
  • ยังไม่ทำ Monitor CPU/Memory ของ process — ยังไม่มี metric endpoint ใดๆ
  • 🔴 ความเสี่ยงสูงสุด ไม่มี process crash guard — ยังไม่มี uncaughtException/unhandledRejection handler
  • ยังไม่ทำ จัดการเคส "ยิง request ออกไปแล้วปลายทางไม่ได้รับจริง" — มี idempotency พื้นฐานที่ webhook แล้ว แต่ยังไม่มี retry policy ที่เป็นระบบ
  • ยังไม่ทำ ระบุ deploy spec ให้ชัดเจน (จำนวน instance, CPU/RAM, restart policy)

📊 4. การประเมิน/ทดสอบโหลด (Capacity Planning & Load Test)

4.1 ข้อมูลตั้งต้นที่ต้องรู้

  • จำนวนผู้ใช้แต่ละกลุ่ม: นักเรียน / สต๊าฟ / ผู้ปกครอง
  • จำนวน request ต่อคนต่อมื้อ — เด็กกินข้าวเช้า: อย่างน้อย 2 req (ข้าว+น้ำ) + ขนม 1 req รวม ~3 req; มื้อกลางวัน: เด็กจำนวนมากลงพักพร้อมกัน (สมมติ 2,000 คน) ต้องเช็คว่าลงพร้อมกันจริงหรือแบ่งรอบ
  • เวลาที่แต่ละคนใช้ต่อ transaction (จับเวลาจริงหน้างาน)
  • ช่วงเวลาที่เกิด peak/normal spike (เช่น ตอนพักเที่ยง)
  • Capacity ทางกายภาพอ้างอิง: โรงเรียนรัฐ/เอกชนทั่วไปรองรับพักพร้อมกันได้ ~800–1,200 คน

4.2 ตัวอย่าง setup ที่ต้องคำนวณโหลดต่อเครื่อง

  • สมมติ 15 ร้าน → POS 15 เครื่อง (แบ่ง ม.ต้น / ม.ปลาย)
  • เด็ก ~800 req กระจายลงกี่เครื่อง → คำนวณโหลดเฉลี่ยต่อเครื่อง POS 1 เครื่อง
  • เข้าใจ flow ว่าแต่ละ request (1 การสั่งซื้อ/1 การสแกนบัตร) ทำงานผ่าน service อะไรบ้าง

4.3 Load test scenarios

  • Baseline: 300 req/นาที (ตรงกับ rate limit จริงที่ตั้งไว้)
  • ทดสอบที่ระดับ 1x, 2x, 5x, 6x ของ baseline
  • เก็บ timer log: 1 request ใช้เวลากี่วินาที (p50/p95/p99)
สถานะ: ยังไม่มี load test script ในโปรเจกต์ (ไม่พบ k6/artillery/autocannon) — ต้องสร้างใหม่ทั้งหมด

🚨 5. Incident Response

  • ยังไม่ทำ เมื่อเกิดเหตุ ต้องหาได้ว่า user ไหน, start date / end date ของช่วงที่มีปัญหา
  • ยังไม่ทำ จัดทำ Incident Report ตอบ 3 คำถามเสมอ: 1) ทำไมระบบล่ม (root cause) 2) แก้ปัญหาอย่างไร (remediation) 3) มี preventive measure หรือไม่
  • ยังไม่ทำ ทำ report เป็น HTML พร้อมกราฟ เพื่อใช้คุยกับทีม support/ภายนอกได้ง่าย

📈 6. Logging & Observability

  • ยังไม่ทำ ประเมินว่า logging ปัจจุบัน (in-system + file) เพียงพอหรือไม่
  • ยังไม่ทำ เขียน test case ให้ครอบคลุม flow สำคัญ (เติมเงิน, หักเงิน, เชื่อม hardware)
  • ยังไม่ทำ ทำระบบ Track/Monitor performance ไว้ย้อนดูเมื่อเกิดเหตุการณ์ผิดปกติ
  • ยังไม่ทำ Log ต้องมี stack trace + error code ในระดับ API ที่ระบบเราเรียกใช้ (รวม 3rd-party API)

🎯 7. ลำดับที่ควรเริ่มทำ (Priority)

เกณฑ์จัดลำดับ: อะไรที่ ยังไม่มีเลย + เสี่ยงทำระบบล่มทั้งระบบ มาก่อน อะไรที่ มีฐานอยู่แล้วแค่ต้องต่อยอด มาทีหลัง

🔴 P0 — ทำก่อนสุด (ช่องโหว่จริง ยังไม่มีการป้องกันเลย)

  1. เพิ่ม process.on('uncaughtException'/'unhandledRejection') guard ใน server.ts — ตอนนี้ error จุดเดียวฆ่าทั้ง process ได้ กระทบทุกร้าน/ทุกเครื่อง POS พร้อมกัน
  2. เขียน automated test สำหรับ wallet/topup/pos-checkout รวมเคส concurrent request — ตอนนี้ไม่มี test เงินเลยแม้แต่ไฟล์เดียว
  3. เขียน deploy spec เป็นเอกสาร (instance/CPU/RAM/restart policy)

🟠 P1 — เตรียม capacity/load test

  1. เก็บจำนวน user จริง (นักเรียน/สต๊าฟ/ผู้ปกครอง) + เวลาต่อ transaction จากหน้างาน
  2. สร้าง load test script (k6/autocannon) ยิงที่ 300 req/min แล้วขยับเป็น 1x/2x/5x/6x
  3. เพิ่ม CPU/Memory metric endpoint เพื่อดูผลตอนรัน load test

🟡 P2 — VA scan / SQLi / SLA formalize

  1. รัน VA scan + SQLi scan อย่างเป็นทางการเพื่อ "ยืนยัน" สิ่งที่โครงสร้างโค้ดทำถูกอยู่แล้ว
  2. ตั้งเกณฑ์ SLA 1 วิ ให้เป็น automated check (เช่น alert ถ้า p95 > 1s)

🔵 P3 — Incident readiness / observability

  1. ทำ retry policy ที่เป็นระบบสำหรับ outbound request
  2. ทำ HTML incident report template + graph
  3. พิจารณาต่อ log ปัจจุบันเข้า dashboard กลาง ถ้าทีมเริ่มเจอ incident บ่อยจนอ่าน log ไฟล์ไม่ทัน

app.ts เรียกทั้ง timerMiddleware และ logging (จาก logger.ts) ติดกัน — ทั้งคู่ derive request-id/start-time แยกกันคนละชุด (requestId vs requestID) และ log start/end ของทุก request ซ้ำสองรอบ ตรวจแล้วพบว่าทุก controller (30 ไฟล์) อ่านแค่ requestId จาก timerMiddleware เท่านั้น — logging ทำงานซ้ำโดยไม่มีใครใช้ผลลัพธ์เลย แต่ต้องระวังตอนลบเพราะ logging พ่วง .use(ip({headersOnly:true})) มาด้วย ซึ่ง RateLimitMiddleware ใช้ ctx.ip เป็น fallback หา client IP

ยืนยันว่าไม่พังหลังแก้

  • tsc --noUnusedLocals --noUnusedParameters — เหลือ error เดียวคือ type mismatch เดิมใน RateLimitMiddleware.ts (รู้จักอยู่แล้วจากก่อนหน้า ไม่เกี่ยวกับ dead code รอบนี้)
  • bun test — 30 pass (เพิ่มจาก 25 เพราะมี crypto.test.ts จากรอบก่อน), 3 fail เดิมที่ไม่เกี่ยวข้อง (schema drift เก่า)
  • บูต server จริง — boot สำเร็จ, 207 routes เท่าเดิม, /health และ /docs ตอบ 200 ปกติ

🚀 9. Deploy Infrastructure — ย้ายจาก Railway มา self-host EC2 (7 ก.ค. 2026)

เลิกใช้ Railway เป็นที่ deploy backend-bun ย้ายมารัน Docker เองบน EC2 (Amazon Linux 2023) ต่อผ่าน Termius ทั้งหมด แยก 2 environment คนละ container/พอร์ตบนเครื่องเดียวกัน

เสร็จแล้ว
Build locally + SFTP แทน git clone/pull บนเซิร์ฟเวอร์

เดิม Dockerfile ต้อง build จาก root ของ monorepo (ต้องพึ่ง bun.lock + shared/) เปลี่ยนเป็น: build dist/server.js ด้วย bun run build บนเครื่อง local ก่อน แล้วลากแค่ Dockerfile, deploy.sh, dist/, .env.prod, .env.uat ขึ้นเซิร์ฟเวอร์ผ่าน SFTP — image เหลือแค่ COPY dist ./dist ไม่ต้อง bun install ในเซิร์ฟเวอร์เลย เบากว่าเดิมมาก และไม่ต้องพึ่งทั้ง repo อีกต่อไป

deploy.sh รองรับ 2 environment บนเครื่องเดียวกัน

รับ argument prod/uat → เลือก container name + env file ให้อัตโนมัติ แล้วดึงค่า PORT จากไฟล์ env มาทำ -p PORT:PORT เอง (ไม่ hardcode) เพื่อกันปัญหาพอร์ตข้างใน container กับพอร์ตที่ Docker forward ไม่ตรงกัน

Seed ข้อมูลทดสอบเข้า UAT DB

แก้แล้ว
บั๊ก/ปัญหาที่เจอระหว่างทาง
  • Docker ไม่ได้ติดตั้งบน EC2 — เครื่องเป็น Amazon Linux 2023 ไม่ใช่ Ubuntu ที่วางแผนไว้แต่แรก ติดตั้งด้วย dnf install docker + เพิ่ม ec2-user เข้า group docker
  • Port ชนกันระหว่าง prod/uat — ตั้ง PORT ใน .env.uat ซ้ำกับ .env.prod (19510 ทั้งคู่) ทำให้ Docker error "port is already allocated" ตอน start container ที่สอง
  • UAT DB password authentication failed — สลับ username เป็น isb-uat ทั้งที่ role นี้ไม่มีอยู่จริงบน Postgres server ทำให้ container ขึ้น crash loop ตลอด แก้กลับไปใช้ user isb เดิม + แยก database ชื่อ isb-uat แทนแยกที่ user
  • Local dev server ตอบ 500 เพราะ DATABASE_URL ค้างของเก่าใน memory — process bun run --watch src/server.ts ที่รันมาก่อนหน้าจะเปลี่ยน .env ยังพยายามต่อ localhost:5432 เก่าอยู่ ค้าง 10 วิ (connect_timeout) แล้ว throw เป็น 500 — แก้ด้วยการ restart dev process ให้อ่าน env ใหม่
Mediumค้างอยู่ — ต้องทำก่อน go-live จริง
Secret/credential ยังเป็นค่า TEMP ใน .env.prod

JWT_SECRET (ต้องตรงกับ FastAPI SECRET_KEY), PYMT_BASE_URL/PYMT_MERCHANT_TOKEN/PYMT_WEBHOOK_SECRET (ต้องเป็น production credential จริงจาก BAY ไม่ใช่ของ UAT sandbox) ยังไม่ได้กรอกค่าจริง — ตอนนี้ deploy ได้แค่ระดับ "โครงรันติด" (skeleton) เท่านั้น auth ข้าม backend และ payment จริงยังใช้ไม่ได้

💡 ค้นพบสำคัญระหว่างทาง: กลไก apply schema จริงของระบบนี้คือ ensureSchema() ที่รันตอน server boot ทุกครั้ง (raw SQL ALTER TABLE ADD COLUMN IF NOT EXISTS แบบ idempotent) ไม่ใช่ drizzle-kit migrate — อธิบายได้ว่าทำไม migration 0001-0003 ใน drizzle/meta/_journal.json ไม่เคยถูกอัปเดตให้ตรงกับไฟล์ .sql ที่มีอยู่จริง (คนละกลไกกัน ไม่ใช่บั๊ก แค่ทีมไม่ได้พึ่ง drizzle-kit migrate จริงจังตั้งแต่แรก) — schema จริงถูก sync อยู่เสมอผ่าน ensureSchema ทุกครั้งที่ server start ไม่ว่า drizzle journal จะตรงหรือไม่ก็ตาม

📖 สรุปศัพท์เทคนิค

คำศัพท์ที่ใช้ในเอกสารนี้ เรียงตามหมวดที่พบมากสุดก่อน พร้อมคำอธิบายแบบเข้าใจง่าย

ความปลอดภัย (Security)

คำศัพท์คำอธิบาย
VA scan
(Vulnerability Assessment)		การสแกนหาช่องโหว่ความปลอดภัยทั้งระบบแบบอัตโนมัติ ทั้งตัว API เอง, library/dependency ที่ใช้, และ container image — เหมือนตรวจสุขภาพระบบว่ามีจุดอ่อนตรงไหนบ้าง
SQL Injectionการโจมตีโดยแทรกคำสั่ง SQL ปลอมเข้าไปใน input (เช่น ช่อง login) เพื่อหลอกให้ฐานข้อมูลรันคำสั่งที่ไม่ควรรัน — ป้องกันได้ด้วยการใช้ ORM/parameterized query แทนการต่อ string SQL เอง
CORS(Cross-Origin Resource Sharing) กฎที่เบราว์เซอร์ใช้ควบคุมว่าเว็บไซต์ไหนมีสิทธิ์เรียก API ของเราได้บ้าง ถ้าตั้งเป็น "อนุญาตทุกเว็บ" (wildcard) จะเสี่ยงให้เว็บปลอมเรียก API เราได้
Webhookกลไกที่ระบบภายนอก (เช่นธนาคาร) ยิง request มาแจ้งระบบเราอัตโนมัติเมื่อมีเหตุการณ์เกิดขึ้น (เช่น "จ่ายเงินสำเร็จแล้ว") ต่างจาก API ปกติที่เราต้องเป็นฝ่ายถามไปก่อน
Signature verificationการตรวจ "ลายเซ็นดิจิทัล" ที่แนบมากับ request เพื่อพิสูจน์ว่า request นั้นมาจากต้นทางจริง (เช่นธนาคาร) ไม่ใช่ของปลอม
Constant-time compare
(timing-safe equal)		วิธีเปรียบเทียบค่าลับ (เช่น password, API key) แบบที่ใช้เวลาเท่ากันเสมอไม่ว่าค่าจะตรงกันแค่ไหน — ป้องกัน "timing attack" ที่คนร้ายอาจเดาค่าลับได้จากการจับเวลาตอบสนอง
Fail-closed / Fail-openเวลาระบบเจอสถานการณ์ผิดปกติ (เช่น ลืมตั้งค่าที่จำเป็น) จะเลือก "ปิดกั้นไว้ก่อน" (fail-closed = ปลอดภัยกว่า แต่ระบบอาจใช้งานไม่ได้ชั่วคราว) หรือ "ปล่อยผ่านไปก่อน" (fail-open = ใช้งานต่อได้ แต่เสี่ยงด้านความปลอดภัย)
JWT(JSON Web Token) โทเค็นยืนยันตัวตนแบบเข้ารหัส ใช้แทนการส่ง username/password ซ้ำทุกครั้ง ระบบเราใช้แนบไปกับ header Authorization: Bearer <token>
Dependency / secret scanningเครื่องมืออัตโนมัติที่คอยตรวจว่า library ที่โปรเจกต์ใช้มีช่องโหว่ที่เพิ่งถูกประกาศหรือไม่ หรือมีการเผลอ commit รหัสลับ (secret) ติดเข้าไปในโค้ดหรือเปล่า

ธุรกรรมการเงินและฐานข้อมูล

Incrementการบวก/ลบยอดจากค่าปัจจุบัน (เช่น ยอดใหม่ = ยอดเก่า + จำนวนเงิน) แทนที่จะ "set" ยอดใหม่ทับตรงๆ — สำคัญมากเพราะถ้ามี 2 request มาพร้อมกัน การ set ทับจะทำให้ยอดหนึ่งหายไป แต่การ increment ที่ล็อกแถวไว้จะปลอดภัยกว่า
Row lock / Resource lock
(SELECT ... FOR UPDATE)		คำสั่งฐานข้อมูลที่ "ล็อก" แถวข้อมูลไว้ระหว่างทำธุรกรรม ไม่ให้ request อื่นมาอ่าน/แก้ไขแถวเดียวกันพร้อมกัน ป้องกันยอดเงินผิดพลาดเวลามีคนเติม/หักเงินพร้อมกันหลายคน
Idempotency / idempotentคุณสมบัติของ operation ที่ทำซ้ำกี่ครั้งก็ได้ผลลัพธ์เหมือนเดิม เช่น webhook ที่ธนาคารอาจยิงซ้ำโดยไม่ตั้งใจ ถ้าระบบเรา idempotent จะไม่เผลอเติมเงินซ้ำ
Transaction (DB)กลุ่มคำสั่งฐานข้อมูลที่ต้อง "สำเร็จทั้งหมดหรือไม่สำเร็จเลย" (all-or-nothing) เช่น หักเงินฝั่งหนึ่งพร้อมบวกเงินอีกฝั่งพร้อมกัน ถ้าขั้นตอนใดพลาด ทุกอย่างจะถูกย้อนกลับ (rollback) ไม่ทิ้งข้อมูลครึ่งๆ กลางๆ

ความทนทานของระบบ (Resilience)

Timeoutเวลาสูงสุดที่ยอมรอคำตอบจากปลายทาง ก่อนจะตัดการเชื่อมต่อแล้วถือว่าล้มเหลว — ป้องกันไม่ให้ request ค้างรอไม่มีที่สิ้นสุดเวลาปลายทางไม่ตอบ
Process crash guard
(uncaughtException /
unhandledRejection)		กลไกดักจับ error ที่ "หลุด" ออกมาโดยไม่มีใคร catch ไว้ ถ้าไม่มีกลไกนี้ error แบบนี้จะทำให้โปรแกรมทั้งตัว (process) ปิดตัวลงทันที กระทบผู้ใช้ทุกคนที่กำลังใช้งานอยู่ ณ ขณะนั้น
CPU/Memory monitoringการเฝ้าระวังว่าเครื่อง/โปรแกรมใช้ทรัพยากร (CPU, RAM) มากแค่ไหนแบบเรียลไทม์ เพื่อรู้ล่วงหน้าก่อนที่เครื่องจะทำงานหนักเกินจนล่ม
Retry policy /
Exponential backoff		นโยบายลองส่ง request ใหม่อัตโนมัติเมื่อครั้งแรกล้มเหลว โดย "exponential backoff" หมายถึงเพิ่มเวลารอระหว่างการลองแต่ละครั้งขึ้นเรื่อยๆ (เช่น รอ 1s, 2s, 4s, 8s) ไม่ให้ยิงถี่จนซ้ำเติมปัญหา
Deploy specเอกสารที่ระบุรายละเอียดการติดตั้งใช้งานจริง เช่น รันกี่ instance, แต่ละเครื่องมี CPU/RAM เท่าไหร่, มีนโยบาย restart อัตโนมัติเมื่อพังหรือไม่

โหลดและการสเกล (Load & Scaling)

Rate limiterตัวจำกัดจำนวน request ที่ยอมให้ผ่านต่อช่วงเวลา (เช่น 300 request/นาที ต่อ IP) เพื่อกันไม่ให้คนใดคนหนึ่งยิงถล่มจนระบบล่ม
In-memory storeการเก็บข้อมูล (เช่น ตัวนับของ rate limiter) ไว้ในหน่วยความจำ (RAM) ของโปรแกรมเอง แทนที่จะเก็บในฐานข้อมูล — เร็วมาก แต่มีข้อเสียคือถ้ารันหลาย instance แต่ละตัวจะมีข้อมูลแยกกันคนละชุด ไม่รู้จักกัน
Instance / Processหนึ่งชุดของโปรแกรมที่กำลังรันอยู่ (เหมือนเปิดแอปเดียวกันหลายหน้าต่างพร้อมกัน) — เวลา "scale-out" คือการเพิ่มจำนวน instance เพื่อรองรับผู้ใช้มากขึ้น
Scale-outการเพิ่มจำนวนเครื่อง/instance ที่รันแอปพลิเคชันเดียวกันขนานกัน เพื่อรองรับโหลด/จำนวนผู้ใช้ที่มากขึ้น (ตรงข้ามกับ "scale-up" ที่คืออัปเกรดเครื่องเดิมให้แรงขึ้น)
Redisฐานข้อมูลแบบ in-memory ความเร็วสูง มักใช้เก็บข้อมูลที่ต้อง "ใช้ร่วมกัน" ระหว่างหลาย instance (เช่น ตัวนับ rate limiter) เพราะเร็วกว่าฐานข้อมูลหลักและทุก instance เห็นข้อมูลชุดเดียวกัน
Load testการจำลองผู้ใช้จำนวนมากยิง request เข้าระบบพร้อมกัน เพื่อดูว่าระบบรับไหวแค่ไหนก่อนจะเจอสถานการณ์จริง (เช่น k6, Artillery, Autocannon คือเครื่องมือที่ใช้ทำ)
p50 / p95 / p99ตัวเลขสถิติเวลาตอบสนอง — p95 คือ "95% ของ request ตอบกลับเร็วกว่าเวลานี้" ใช้ดูภาพรวมได้แม่นกว่าค่าเฉลี่ยเฉยๆ เพราะไม่ถูกกลบด้วย request ส่วนน้อยที่เร็วมากๆ

การตรวจสอบและกู้คืนเหตุการณ์ (Observability & Incident)

SLA(Service Level Agreement) ข้อตกลง/เป้าหมายระดับบริการที่ตั้งไว้ล่วงหน้า เช่น "ทุก API ต้องตอบกลับภายใน 1 วินาที" ใช้เป็นเกณฑ์วัดว่าระบบทำงาน "ดีพอ" หรือไม่
CI (Continuous Integration)กระบวนการอัตโนมัติที่รันตรวจสอบโค้ด (เช่น test, scan ความปลอดภัย) ทุกครั้งที่มีการแก้โค้ด ก่อนจะปล่อยขึ้นใช้งานจริง
APM
(Application Performance Monitoring)		เครื่องมือ/แดชบอร์ดกลางที่รวบรวม log และ performance ของระบบไว้ที่เดียว ให้ค้นหา/ดูย้อนหลังได้ง่ายเวลาต้องสืบว่าเกิดอะไรขึ้น (เช่น Grafana, ELK)
Incident reportรายงานสรุปเหตุการณ์ระบบขัดข้อง ที่ควรตอบ 3 คำถามเสมอ: เกิดจากอะไร (root cause), แก้อย่างไร (remediation), จะป้องกันไม่ให้เกิดซ้ำได้อย่างไร (preventive measure)
Priority P0 – P3ระดับความสำคัญของงาน เรียงจากเร่งด่วน/กระทบมากที่สุด (P0) ไปจนถึงรอได้ (P3) — ใช้จัดลำดับว่าควรลงมือทำอะไรก่อน-หลัง

SSH(Secure Shell) โปรโตคอลเข้ารหัสสำหรับต่อเข้าเทอร์มินัลของเครื่องอื่นระยะไกล ใช้ยืนยันตัวตนด้วย key แทน password ได้
SFTP(SSH File Transfer Protocol) วิธีโอนไฟล์ระหว่างเครื่องผ่านการเชื่อมต่อ SSH เดียวกัน — ใช้แทนการ git pull บนเซิร์ฟเวอร์ในโปรเจกต์นี้
Docker image / containerImage คือ "พิมพ์เขียว" ของแอป (โค้ด + runtime) ที่ build ไว้ครั้งเดียว ส่วน container คือตัวที่รันจริงจาก image นั้น — รันกี่ container จาก image เดียวกันก็ได้ (เช่น prod/uat)
Docker build contextโฟลเดอร์ที่ Docker ใช้เป็นขอบเขตตอน build image (คำสั่ง COPY ใน Dockerfile อ่านไฟล์จากขอบเขตนี้เท่านั้น) — เปลี่ยนจาก root ทั้ง monorepo มาเป็นแค่ backend-bun/ ทำให้ build เร็วและง่ายขึ้นมาก
--env-fileFlag ของ docker run ที่ระบุไฟล์ env (เช่น .env.prod) ให้ inject เป็น environment variable เข้า container ตอนรัน โดยไม่ต้องฝังค่าไว้ใน image
Amazon Linux / dnfAmazon Linux คือ Linux distro ของ AWS เอง (ต่างจาก Ubuntu) ใช้ dnf เป็นตัวจัดการ package แทน apt ของ Ubuntu — คนละคำสั่งติดตั้งโปรแกรมกันคนละ distro
Docker group / newgrpผู้ใช้ต้องอยู่ใน group docker ถึงจะรันคำสั่ง docker ได้โดยไม่ต้อง sudo — newgrp docker คือคำสั่งให้ session ปัจจุบันรับสิทธิ์ group ใหม่ทันทีโดยไม่ต้อง logout/login
ensureSchema()ฟังก์ชันในโค้ดที่รันตอน server start ทุกครั้ง คอย "ปะ" schema ของ DB ให้ตรงกับที่โค้ดคาดหวัง (เพิ่ม column/table/index ที่ยังไม่มี) แบบ idempotent — เป็นกลไกจริงที่ใช้ sync schema ของโปรเจกต์นี้ ไม่ใช่ drizzle-kit migrate
pg_dump / pg_restoreเครื่องมือมาตรฐานของ Postgres สำหรับ "ส่งออก" ฐานข้อมูลทั้งลูกเป็นไฟล์ (pg_dump) แล้ว "นำเข้า" กลับเข้า DB อื่น (pg_restore) ใช้ตอน copy ข้อมูลจาก local ไปใส่ UAT
bcryptอัลกอริทึมเข้ารหัสรหัสผ่านทางเดียว (hash) ที่ตั้งใจให้คำนวณช้าโดยตั้งใจ (ปรับด้วยค่า "cost") เพื่อกันการ brute-force เดารหัสผ่าน — ระบบนี้ใช้ cost 10 ผ่าน Bun.password.hash()

Code Cleanup / Dead Code

Dead codeโค้ดที่เขียนไว้แต่ไม่มีที่ไหนเรียกใช้งานจริงแล้ว (เช่น ฟังก์ชันเก่าที่ถูกแทนที่ด้วยตัวใหม่แต่ลืมลบ) เก็บไว้ไม่มีประโยชน์ มีแต่ทำให้โค้ดอ่านยากขึ้นและเสี่ยงสับสนว่าอันไหนคือของจริง
Unused import/variableการ import module หรือประกาศตัวแปรมาแล้วไม่เคยใช้ในไฟล์นั้นเลย — ไม่กระทบการทำงานของโปรแกรม แต่เป็น "ขยะ" ที่ทำให้ไม่รู้ว่าไฟล์นี้ต้องพึ่งอะไรจริงๆ บ้าง
Orphan fileไฟล์ที่มีอยู่ในโปรเจกต์แต่ไม่มีไฟล์ไหนเลย import เข้าไปใช้ — เหมือนโค้ดที่ถูกลืมทิ้งไว้ ปลอดภัยที่จะลบเพราะไม่มีอะไรอ้างอิงถึง
Barrel (re-export)ไฟล์ที่ทำหน้าที่รวบรวม export จากหลายไฟล์มาไว้ที่จุดเดียว (เช่น export * from "./a"; export * from "./b";) เพื่อให้ที่อื่น import จากจุดเดียวได้ง่ายขึ้น — แต่ถ้าไม่มีใคร import ผ่าน barrel นี้เลย มันก็กลายเป็น orphan file ไปด้วย
@deprecatedป้ายกำกับในคอมเมนต์ที่นักพัฒนาติดไว้บนโค้ดเพื่อบอกว่า "เลิกใช้อันนี้แล้ว มีของใหม่มาแทน" เป็นสัญญาณว่าถ้าเจอโค้ดไหนไม่ได้ใช้อยู่แล้วด้วย มักจะลบออกได้ปลอดภัย
tsc (TypeScript compiler)โปรแกรมที่ตรวจสอบชนิดข้อมูล (type) ในโค้ด TypeScript — flag --noUnusedLocals/--noUnusedParameters สั่งให้มันช่วยหา import/ตัวแปร/parameter ที่ไม่ได้ใช้ให้อัตโนมัติ แทนที่จะไล่หาด้วยตาเอง
Lockfile
(เช่น bun.lock)		ไฟล์ที่บันทึกเวอร์ชันที่แน่นอนของทุก dependency (รวมถึง dependency ของ dependency) ที่โปรเจกต์ติดตั้งไว้จริง เพื่อให้ทุกเครื่อง/ทุกครั้งที่ install ได้เวอร์ชันเดียวกันเป๊ะ ต้องอัปเดตไฟล์นี้ทุกครั้งที่แก้ package.json
