วัตถุประสงค์

ระบบชำระเงินและบริหารงานครบวงจรสำหรับสหกรณ์และโรงอาหารของโรงเรียนนานาชาติ ผู้ปกครองและบุคลากรเติมเงินเข้ากระเป๋า campus card นักเรียนและบุคลากรใช้จ่ายที่หน้า POS ผู้ดูแลระบบจัดการผู้ใช้ แผนก คืนสินค้า ตั้งค่าระบบ และดู report ระบบยังขับเคลื่อนหน้าจอที่ 2 (Customer Display) เพื่อให้ลูกค้าเห็นรายการที่ซื้อแบบโปร่งใส

🏪 โมดูล Canteen vs Store

ระบบ POS แบ่งเป็น 2 โมดูลหลักที่ทำงานคนละแบบและมี business rule ต่างกัน การออกแบบใดๆ ต่อจากนี้ ต้องเช็คก่อนว่ากำลังทำที่โมดูลไหน

หัวข้อ 🍜 Canteen (โรงอาหาร) 🛒 Store / Coop (สหกรณ์ ร้านหนังสือ Sports)
ประเภทสินค้า อาหาร เครื่องดื่ม ขนม (ผัดไทย, ก๋วยเตี๋ยว, ลาเต้ ฯลฯ) เครื่องเขียน เครื่องแบบ หนังสือ อุปกรณ์กีฬา
หน้าจอ POS /canteen — Grid card สี เน้นการกดเร็ว /store — Card + Sheet cart side panel
วิธีชำระเงิน เงินสด · บัตร campus card (wallet) · QR PromptPay เงินสด · บัตร · QR · บัตร EDC · เบิกแผนก
Bundle / SET ❌ ไม่มี (อาหารขายเป็นรายชิ้น) ✅ มี (เช่น "ชุดเครื่องแบบ" 1 SET = เสื้อ+กางเกง+ป้ายชื่อ)
การคืนสินค้า ❌ ไม่มี (อาหารคืนไม่ได้) ✅ มี — Returns + Return History + bundle restore stock ครบทุก sub-SKU
Department Billing ❌ ไม่มี ✅ มี (เฉพาะร้าน category="coop") — staff เบิกของผ่านงบแผนก
การเบิก (Requisition) ❌ ไม่มี ✅ มี — /store/requisition ติดตามการเบิกข้ามร้าน
โครงสร้างร้าน หลาย stall ภายในโซนเดียว (Thai Kitchen, Western, Drinks) ร้านเดี่ยว (Sports, Bookstore, Coop) แต่ละร้านมี shop_id ของตัวเอง
Area Manager มี — manager ที่ไม่ผูก shop_id เห็นทุก stall ในโซนนี้ มี — manager ที่ไม่ผูก shop_id เห็นทุก store
เมนูจัดการสินค้า /canteen/products (CanteenMenuPage) — เน้นรูป + สี /store/management/{shopId} — เน้น SKU + price panel
หน้าหลัก Manager /canteen/management + /canteen/management/{shopId} /store/management + ShopDetail per shop
รายงาน /canteen/reports — สรุปยอดและ best-selling อาหาร /store/reports — รวม Returns, Stock Card, Sales Summary, Sales by Item ด้วย

หลักการเข้าถึงโมดูล (Module Access)

  • Admin: เห็นทั้ง 2 โมดูล ไม่จำกัด
  • Manager/Cashier ที่ผูก shop_id: เห็นเฉพาะโมดูลที่ shop นั้นสังกัด (canteen หรือ store)
  • Area Manager (ไม่มี shop_id): มี shop_module = canteen หรือ store → เห็นเฉพาะโมดูลของตัวเอง
  • RequireModule guard: ป้องกัน manager โซน canteen เข้า /store/* หรือกลับด้าน

👥 รายชื่อบทบาท (Roles)

บทบาทกลุ่มเป้าหมายสิทธิ์การเข้าถึง
Adminฝ่าย IT/ปฏิบัติการของโรงเรียนตั้งค่าระบบทั้งหมด ดูข้อมูลและรายงานทุกร้าน
Managerผู้จัดการร้าน (ร้านเดียวหรือทั้งโซน)POS + Reports + Users (เฉพาะร้านตัวเอง) ไม่เห็นการตั้งค่าระบบ
Cashierพนักงานคิดเงินเฉพาะ POS — ขาย คืน ยกเลิก ในร้านตัวเอง
Parentผู้ปกครองนักเรียนหน้าครอบครัว เติมเงิน ดูยอด ดูประวัติการใช้จ่ายของลูก
Studentนักเรียนดูยอดและประวัติของตัวเองเท่านั้น
Staffครู/บุคลากรกระเป๋าเงินส่วนตัว + การเบิกของผ่านแผนก

Admin ฟังก์ชันของผู้ดูแลระบบ

การจัดการผู้ใช้

  • สร้าง/แก้ไข/ปิดบัญชีผู้ใช้ทุกบทบาท สูง
  • นำเข้าข้อมูลแบบ Bulk จาก PowerSchool (CSV) — นักเรียน ผู้ปกครอง บุคลากร
  • รีเซ็ตรหัสผ่าน กำหนดบทบาทและร้านสังกัด
  • เชื่อมผู้ปกครอง ↔ นักเรียน ผ่าน family_code
  • อนุมัติการสลับบทบาทสำหรับผู้ใช้ที่มีหลายบทบาท (เช่น manager + cashier)

กระเป๋าเงินและบัตร

  • ปรับยอดเงินในกระเป๋าใดก็ได้ (บันทึก audit log) สูง
  • โอนระหว่างกระเป๋า (เฉพาะ admin เท่านั้น — ผู้ใช้ทำไม่ได้)
  • จัดการการผูกบัตร RFID/NFC กับกระเป๋าเงิน
  • กำหนด negative_credit_limit ต่อผู้ใช้ (วงเงิน overdraft)
  • workflow เปลี่ยนบัตร (ปิดบัตรหาย เปิดบัตรใหม่)

การเบิกผ่านแผนก (Department Billing)

  • สร้าง/ใส่งบให้แต่ละแผนก สูง
  • ปรับยอดงบแผนก พร้อม audit trail
  • เบิกของผ่านสหกรณ์โดยใช้งบแผนก (ไม่มีเงินสดเปลี่ยนมือ)

แคตตาล็อกและการตั้งราคา

  • จัดการสินค้าในร้าน (เมนูโรงอาหาร สินค้าสหกรณ์ ร้านหนังสือ)
  • ตั้ง price panel (ราคาแยกตามร้าน)
  • จัดการ Bundle/SET (1 SKU = หลาย sub-SKU)
  • จัดการหมวดหมู่สินค้าต่อร้าน

รายงาน (Admin: เห็นทุกร้าน)

  • Sales Report (ตามช่วงวันที่ ร้าน สินค้า) สูง
  • Top-Selling Products สินค้าขายดี
  • Sales by Payment Method แยกตามวิธีชำระเงิน
  • Stock Report สต็อกทุกร้าน
  • Return Report การคืนสินค้า
  • Stock Card รายการเคลื่อนไหวต่อ SKU (เปิด-ปิด)
  • Sales Summary รายงานต่อใบเสร็จพร้อมแยกวิธีชำระเงิน ใหม่
  • Sales by Item รายงานต่อรายการสินค้าในใบเสร็จ ใหม่
  • ทุก report ส่งออกเป็น CSV / Excel / PDF (พร้อมโลโก้และชื่อโรงเรียน)

การตั้งค่าระบบ

  • ข้อมูลโรงเรียน (ชื่อ ที่อยู่ เลขผู้เสียภาษี โลโก้ Cover) ปานกลาง
  • จัดการรูปภาพ Customer Display (สูงสุด 10 รูป 2 MB JPG/PNG)
  • ดู Audit log (บันทึก operation สำคัญที่แก้ไม่ได้)
  • Sync กับ PowerSchool ผ่าน cron

Manager ฟังก์ชันของผู้จัดการ

การใช้งาน POS (เฉพาะร้านที่จัดการ)

  • ใช้งาน POS แบบเดียวกับ cashier (ขาย ยกเลิก คืนสินค้า) สูง
  • รับเงินได้ทุกรูปแบบ: เงินสด, กระเป๋าเงิน, บัตร EDC, QR, เบิกแผนก
  • มีหน้าจอที่ 2 (Customer Display) แสดงรายการให้ลูกค้าเห็น
  • ตัวนับยอดขายแบบ real-time วันนี้

ผู้ใช้ในร้าน

  • เพิ่ม/แก้ไข cashier และ staff ในร้านตัวเอง สูง
  • ไม่สามารถสร้าง admin หรือเลือกร้านอื่นได้
  • รีเซ็ตรหัสผ่านของพนักงานในร้าน

การคืนสินค้าและเปลี่ยนของ (เฉพาะสหกรณ์)

  • คืนเงินพร้อมบันทึก audit
  • คืน bundle จะคืน stock ไปยังทุก sub-SKU อัตโนมัติ
  • ประวัติการคืนสินค้า กรองตามสถานะได้

คลังสินค้า

  • ปรับ stock พร้อมเหตุผล
  • ปฏิบัติการแบบ batch — เก็บใน localStorage 12 ชั่วโมง
  • ติดตามการเบิก (Store Requisition)

รายงาน (Manager: เฉพาะร้านตัวเอง)

  • รายงานเหมือน admin แต่จำกัดที่ shop_id ของ manager
  • Area manager (ไม่มี shop_id) เห็นทุกร้านในโมดูล (โรงอาหารหรือสหกรณ์)

Cashier ฟังก์ชันของพนักงานคิดเงิน

การขายที่ POS

  • ค้นหาสินค้าด้วยชื่อ/บาร์โค้ด/หมวดหมู่ สูง
  • เพิ่มลงตะกร้า กำหนดจำนวน และส่วนลดรายชิ้น
  • แตะบัตร campus card → ระบุสมาชิก → ตัดเงินจากกระเป๋า
  • สร้าง QR PromptPay + เช็คสถานะการชำระอัตโนมัติ
  • รับเงินสดพร้อมคำนวณเงินทอนอัตโนมัติ
  • พิมพ์ใบเสร็จความร้อน (Star Micronics 80mm ภาษาอังกฤษ ฟอนต์ใหญ่)
  • ส่งใบเสร็จทาง email ได้

หน้าจอที่ 2 (Customer Display)

  • Popup window ที่ /customer-display
  • โหมด standby หมุนรูปที่ admin อัพโหลด
  • หน้า Order Review + QR (แบ่ง 60:40) + Processing (โชว์ผู้จ่ายและยอดคงเหลือ) + Success / Failed
  • กลับ standby อัตโนมัติหลัง 5 วินาทีเมื่อเสร็จ

ใบเสร็จและการยกเลิก

  • รายการใบเสร็จ กรองวันที่ + ค้นหาได้
  • พิมพ์ใบเสร็จซ้ำ
  • ยกเลิกใบเสร็จ (ใส่เหตุผล บันทึก audit)

Parent ฟังก์ชันของผู้ปกครอง

หน้าครอบครัว

  • เห็นนักเรียนทุกคนที่ผูกไว้ พร้อมกระเป๋าเงิน สูง
  • เห็นยอดเงินปัจจุบันและรายการล่าสุดของลูกแต่ละคน
  • เห็นรูปและชั้นของลูก (มีนโยบายความเป็นส่วนตัวกำกับ)

การเติมเงิน

  • เติมเงินกระเป๋าตัวเองผ่าน PromptPay QR + Bank Transfer สูง
  • เติมเงินกระเป๋าลูกได้โดยตรง (เป็นการ เติมเงิน ไม่ใช่การ โอน)
  • รับการยืนยันธุรกรรมทาง push / email

ประวัติธุรกรรม (อ่านอย่างเดียว)

  • เห็นว่าลูกซื้ออะไร ที่ไหน เมื่อไหร่ สูง
  • เห็นทุกการเติมเงินเข้ากระเป๋าลูก (วันที่ จำนวน แหล่งที่มา)
  • กรองตามวันที่ ประเภทธุรกรรม ร้าน
  • เห็นรายการสินค้าในใบเสร็จ (เช่น "ผัดไทย ฿55", "ลาเต้เย็น ฿45")
  • ส่งออก PDF เก็บไว้

สิ่งที่ผู้ปกครองทำ ไม่ได้ (ตามนโยบาย)

  • ห้ามโอนเงินจากกระเป๋าตัวเองไปกระเป๋าลูก — ทำได้แค่เติมเงินเท่านั้น
  • ห้ามโอนระหว่างลูก 2 คน — ต้องให้ admin เป็นคนทำ
  • ห้ามจ่ายเงินแทนลูกที่หน้า POS — ลูกต้องแตะบัตรเอง
  • ห้ามขอคืนเงินเอง — ต้องไปที่ผู้จัดการร้าน

Student ฟังก์ชันของนักเรียน

ดูข้อมูลตัวเอง

  • ดูยอดเงินในกระเป๋าตัวเอง
  • ดูธุรกรรมล่าสุด (การซื้อที่โรงอาหาร/สหกรณ์/ร้านหนังสือ)
  • เติมเงินเองไม่ได้ — ต้องผ่านผู้ปกครองหรือ admin
  • โอนเงินไม่ได้ — ปิดตามนโยบาย

Staff ฟังก์ชันของบุคลากร

กระเป๋าเงินส่วนตัว

  • UI เดียวกับผู้ปกครองสำหรับกระเป๋าตัวเอง
  • เติมเงินผ่าน QR / Bank Transfer
  • ดูประวัติของตัวเอง

การเบิกผ่านแผนก (เฉพาะสหกรณ์)

  • เบิกของโดยใช้งบแผนก (ไม่ใช่เงินสด)
  • มียอดงบแผนกและเพดานอนุมัติกำกับ
  • เห็นรายการเบิกของตัวเองและประวัติของแผนก

โหมด Cashier เสริม (ถ้ามี shop_id)

  • Staff ที่ผูกกับ shop_id ใดร้านหนึ่งจะได้สิทธิ์เทียบเท่า cashier
  • เหมาะสำหรับครูช่วยขายที่งานโรงอาหาร

🔒 ข้อกำหนดที่ใช้ร่วมกัน

ความปลอดภัย

  • JWT-based auth พร้อม refresh token
  • Google OAuth (เลือกเปิด ใช้ VITE_GOOGLE_CLIENT_ID)
  • ควบคุมสิทธิ์ตามบทบาททั้งฝั่ง API และ UI
  • Audit log สำหรับ operation สำคัญ (ปรับยอด, ยกเลิกบิล, ปรับยอดแผนก)
  • CORS อนุญาตเฉพาะ Vercel host

การปฏิบัติตามกฎ

  • รูปนักเรียน: แสดงเฉพาะฝั่ง cashier ไม่แสดงบน customer display
  • สอดคล้อง PDPA (ไม่มี PII บนหน้าจอที่ลูกค้าเห็น)
  • Department use ไม่ถูกรวมใน Grand Total ของรายงานการชำระเงิน

ประสิทธิภาพและความเสถียร

  • Self-heal schema migration ตอน deploy production
  • ตัวบอกสถานะ server poll ทุก 15 วินาที
  • ใบเสร็จคงอยู่หลัง pg_attribute verification ของ Railway
  • หน้าต่าง customer display ยังคงอยู่หลัง reload (mirror state ผ่าน localStorage)

ภาษา

  • i18next (en + th) ทั่วทั้งระบบ; ใบเสร็จบังคับเป็นภาษาอังกฤษ (เพราะเป็นโรงเรียนนานาชาติ)
  • หน้าต่าง customer display: ภาษาอังกฤษเท่านั้น

🖥️ Kiosk — ตู้บริการตนเอง (Touch Screen)

แอป Vue 3 แยกต่างหากจาก React หลัก ออกแบบสำหรับตู้หน้าจอสัมผัสที่ตั้งไว้บริเวณทางเข้าโรงอาหาร/สหกรณ์ ให้นักเรียน/บุคลากรเช็คยอด ดูประวัติ และเติมเงินได้เองโดยไม่ต้องต่อคิวที่แคชเชียร์ ไม่มีการ login — ระบุตัวตนด้วยการแตะบัตร RFID หรือใส่รหัสด้วย Numpadเท่านั้น เป็น read-only สำหรับข้อมูลส่วนตัว การเติมเงินทำผ่าน QR ไม่เก็บข้อมูลบัตร

การระบุตัวตน

  • แตะบัตร RFID บนเครื่องอ่านที่ติดอยู่ (วิธีหลัก) สูง
  • ใส่รหัสด้วย Numpad บนหน้าจอ (วิธีสำรอง)
  • กลับสู่หน้า Welcome อัตโนมัติเมื่อไม่มีการใช้งาน 30 วินาที
  • ไม่มีรหัสผ่าน/PIN — บัตรในมือคือการยืนยันตัวตน

เช็คยอดเงิน (BalanceView)

  • แสดง: ชื่อผู้ถือบัตร ยอดปัจจุบัน รายการล่าสุด 3 รายการ สูง
  • ฟอนต์ใหญ่อ่านง่ายในระยะแขน
  • ปุ่ม "เติมเงิน" / "ประวัติทั้งหมด" / "เสร็จสิ้น"
  • ไม่แสดงรูป (เพราะตู้อยู่ในพื้นที่สาธารณะ — privacy)

ประวัติธุรกรรม (TransactionHistoryView)

  • รายการแบ่งหน้า มี icon ตามประเภท (ซื้อ/เติม/โอน/คืน) ปานกลาง
  • เห็นวันที่ ชื่อร้าน และจำนวนเงินในแต่ละบรรทัด
  • Popup ดูรายละเอียดใบเสร็จ (ReceiptPreview)
  • กรองตามวันที่และประเภทธุรกรรม

เติมเงิน (TopUpView)

  • Numpad ใส่จำนวนเงิน (preset: 50, 100, 200, 500) สูง
  • สร้าง PromptPay QR ผ่าน backend /wallets/{id}/topup-qr
  • เช็คสถานะการจ่ายอัตโนมัติ แสดงหน้า success เมื่อยืนยัน
  • ยกเลิกกลับ Welcome ได้ตลอด; QR หมดอายุใน 15 นาที

หน้า Welcome / Idle

  • คำเชิญใหญ่ "แตะบัตรหรือใส่รหัส"
  • มี animation/branding ของโรงเรียน
  • หน้า default หลังหน้าอื่น timeout

ข้อจำกัดในการทำงาน

  • API ที่ใช้เป็น read-only สำหรับยอดและประวัติ ไม่มี endpoint ของ admin
  • Stateless: ไม่เก็บข้อมูลข้าม session ทุกการแตะเริ่มใหม่
  • UI สำหรับ touch: ปุ่มใหญ่ ไม่มี tooltip ไม่ใช้ hover
  • เปลี่ยนภาษา en/th ได้จากหน้า Welcome
  • รันใน fullscreen / kiosk mode (ไม่มี nav bar)
  • Deploy แยกใน Vercel มี URL ของตัวเอง
