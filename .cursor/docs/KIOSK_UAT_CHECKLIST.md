# Kiosk UAT Checklist (Android)

ใช้ก่อนเปิดใช้งานจริงที่โรงเรียน — ทดสอบบน **อุปกรณ์จริง** (NK77 bill acceptor + NFC/barcode reader)

## ก่อนเริ่ม

- [ ] Kiosk `.env` ชี้ API production/staging ที่ถูกต้อง
- [ ] อุปกรณ์เชื่อม Wi‑Fi โรงเรียนได้เสถียร
- [ ] มีนักเรียนทดสอบอย่างน้อย 2 คน (บัตร NFC + student code)
- [ ] มีธนบัตรทดสอบ (20/50/100 บาท) — ไม่ใช่เงินจริงของผู้ใช้ถ้าเป็น staging
- [ ] ทีม support รู้วิธีดู log / wallet transactions (ดู `INCIDENT_RESPONSE_PLAYBOOK.md`)

## 1. ค้นหานักเรียน / สแกนบัตร

| # | ขั้นตอน | ผลที่คาดหวัง | ✓ |
|---|---------|--------------|---|
| 1.1 | สแกน NFC บัตรนักเรียน | แสดงชื่อ + wallet ถูกต้อง | |
| 1.2 | พิมพ์/สแกน `student_code` (barcode) | ค้นหาเจอคนเดียวกับ NFC | |
| 1.3 | บัตรที่ไม่มีในระบบ | แสดงข้อความไม่พบ ไม่ crash | |
| 1.4 | สแกนซ้ำหลัง session จบ | เริ่ม flow ใหม่ได้ปกติ | |

## 2. ตรวจยอด / ประวัติ

| # | ขั้นตอน | ผลที่คาดหวัง | ✓ |
|---|---------|--------------|---|
| 2.1 | เปิดหน้าประวัติ top-up | แสดงรายการจาก `GET /wallets/:id/transactions` | |
| 2.2 | เปรียบเทียบยอดกับ admin/POS | ตรงกัน | |

## 3. เติมเงินสด (Bill acceptor)

| # | ขั้นตอน | ผลที่คาดหวัง | ✓ |
|---|---------|--------------|---|
| 3.1 | เลือกจำนวนเงิน แล้วเริ่มรับธนบัตร | เครื่องรับเงิน, แสดงยอดสะสม | |
| 3.2 | ใส่เงินครบเป้า | **ไม่ต้องกด confirm** — auto success | |
| 3.3 | ยอด wallet หลัง success | เพิ่มตามจำนวนที่ใส่ (ตรวจ API/DB) | |
| 3.4 | ระหว่างมีเงินในเครื่องแล้ว กด back/logout/เปลี่ยนวิธี | **ถูกบล็อก** หรือแสดง modal เตือน | |
| 3.5 | ยกเลิกก่อนใส่เงิน | กลับได้ตามปกติ | |
| 3.6 | ยกเลิกหลังใส่เงินแล้ว | modal เตือน + คืนเงินตาม flow hardware | |
| 3.7 | ใส่เกินจำนวน (overpay) | แสดง overpay UI, คืนส่วนเกินหรือยอมรับตามที่ออกแบบ | |
| 3.8 | ปิดแอปขณะรอ API หลังใส่เงินครบ | เปิดใหม่ → retry pending (`kiosk-pending-cash-topup`) | |
| 3.9 | ปิดแอปแล้วเปิดซ้ำหลัง success | ไม่ credit ซ้ำ — server ใช้ `idempotency_key` คู่กับ pending retry | |

## 4. เติมเงิน QR

| # | ขั้นตอน | ผลที่คาดหวัง | ✓ |
|---|---------|--------------|---|
| 4.1 | สร้าง QR intent | แสดง QR + ref | |
| 4.2 | ชำระสำเร็จ (หรือ simulate confirm) | ยอด wallet เพิ่ม, ประวัติมี `TOPUP` | |
| 4.3 | ชำระซ้ำ / confirm ซ้ำ | credit ครั้งเดียว (idempotent ที่ backend) | |

## 5. เครื่องพิมพ์ / ใบเสร็จ

| # | ขั้นตอน | ผลที่คาดหวัง | ✓ |
|---|---------|--------------|---|
| 5.1 | พิมพ์จากหน้าประวัติ | กดรายการ → `ReceiptPreview` → พิมพ์ thermal 80mm | |
| 5.2 | หลัง top-up สำเร็จ (cash) | **auto-print** บนหน้า success + ปุ่มพิมพ์ซ้ำ; มีเลข `transaction_id` | |
| 5.3 | หลัง top-up สำเร็จ (QR) | auto-print + เลข `transaction_id` จาก `GET /topup/:ref/status` | |

## 6. ความทนทานเบื้องต้น

| # | ขั้นตอน | ผลที่คาดหวัง | ✓ |
|---|---------|--------------|---|
| 6.1 | ตัดเน็ตระหว่าง top-up | แสดง error ที่เข้าใจได้, ไม่ค้าง state | |
| 6.2 | API ช้า (>10s) | มี loading / ไม่กดซ้ำจน duplicate | |
| 6.3 | Idle 30 นาที | session timeout ตามที่กำหนด (ถ้ามี) | |
| 6.4 | รีสตาร์ทเครื่อง | แอป boot ได้, hardware reconnect | |

## 7. ความปลอดภัย / การใช้งาน

- [ ] ไม่แสดง token / internal ID บนหน้าจอผู้ใช้
- [ ] Logout ล้างข้อมูล session บนเครื่อง
- [ ] ไม่มี path ข้าม authentication ของ kiosk API key (ถ้ามี)
